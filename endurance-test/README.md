# Endurance Test Suite

This document describes how to use the endurance test suite.

## Overview

The endurance test suite is designed to run a sampling job repeatedly to ensure the stability and reliability of the system over a long period. It uses [`runn`](https://github.com/k1LoW/runn) to define and execute the test scenarios.

The main test script, `run_endurance_test.sh`, executes the `endr.yml` test scenario every 10 seconds, indefinitely, until stopped.

## Test Content

Each iteration performs one full round-trip against the quantum cloud API.

### 1. Quantum circuit under test

A 2-qubit Bell-state circuit written in OpenQASM 3:

```qasm
OPENQASM 3;
include "stdgates.inc";
qubit[2] q;
bit[2] c;
h q[0];
cx q[0],q[1];
c = measure q;
```

- `job_type`: `sampling`
- `shots`: `1000`
- `transpiler_info`: `qiskit` with `optimization_level: 1`
- `mitigation_info`: none

### 2. Execution steps (`runn` scenario)

The scenario is composed in three layers:
`endr.yml` → `runn-included/sampling-included.yml` → `runn-included/post.yml`.

| Step | What it does |
|---|---|
| `device` | Dumps the target device ID (`DEVICE_ID`) to the log |
| `post` | Submits the job via `POST /jobs` and asserts HTTP `200` |
| `message` | Logs the returned `job_id` with a timestamp |
| `wait` | Polls `GET /jobs/{job_id}` **up to 20 times at 1 second intervals**, waiting until the status becomes one of `succeeded` / `failed` / `canceled` |

### 3. Assertions

After the job reaches a terminal state, the test blocks in `sampling-included.yml` verify the following.

- **status-test**: Final status equals `succeeded`.
- **basic-test**: All of the following hold:
  - `job_id` in the polled response matches the one returned by `POST /jobs`.
  - `name`, `description`, `device_id`, `shots`, `job_type` match the request.
  - The submitted `program` is preserved as a single-element list.
  - `job_info.result.sampling.counts` is present.
  - `execution_time > 0`.
  - Timestamps are monotonically ordered: `submitted_at ≤ ready_at ≤ running_at ≤ ended_at`.
- **transpiler-info-test**: `transpiler_info.transpiler_lib` and `transpiler_options` in the response match what was submitted.
- **transpiler-result-test**: Either `transpile_result` is empty, or `transpiled_program`, `stats`, and `virtual_physical_mapping` are all populated.
- **mitigation-info-test**: `mitigation_info.ro_error_mitigation` matches the request.
- **simulator-info-test**: `simulator_info` is `null` (i.e. the job ran on real hardware).

### 4. Loop behavior (`run_endurance_test.sh`)

- Logs the start time, end time, and elapsed seconds of each iteration.
- Each iteration runs on a **fixed 10-second period**. If the scenario finishes in under 10 seconds, the script sleeps for the remaining time; if it overruns, the next iteration starts immediately without waiting.
- If `runn run` exits with a non-zero code, the log records `runn command failed with exit code N`, but the loop continues.
- The script writes its own PID to `endurance_test.pid`.

## Prerequisites

- [runn](https://github.com/k1LoW/runn) installed
- [Task](https://taskfile.dev/) installed (for running Taskfile.yml commands)
- `bc` installed (used by `run_endurance_test.sh` for sub-second arithmetic; not preinstalled on minimal Linux images such as Amazon Linux). Install with `sudo dnf install -y bc` on Amazon Linux 2023, `sudo yum install -y bc` on Amazon Linux 2, or `sudo apt-get install -y bc` on Debian/Ubuntu.
- A `.env` file in the `endurance-test` directory with the required environment variables

## File Layout

- **`run_endurance_test.sh`**: The shell script that runs the test in a loop.
- **`Taskfile.yml`**: Defines tasks for starting, stopping, and checking the status of the endurance test.
- **`endr.yml`**: The top-level `runn`book. Defines the Bell-state sampling scenario and includes `sampling-included.yml`.
- **`runn-included/sampling-included.yml`**: Defines the assertions for the sampling job and includes `post.yml`.
- **`runn-included/post.yml`**: Handles the job submission (POST) and polls for completion (GET loop).
- **`results/`**: Log files generated by each run (`endurance_test_YYYY_MMDD_HHMM_SS.log`).
- **`endurance_test.pid`**: PID file of the running loop (created on start, removed on stop).

## How to Use

### 1. Create `.env` file

Create a `.env` file in the `endurance-test` directory with the following content:

```bash
USER_API_ENDPOINT="<your-api-endpoint>"
Q_API_TOKEN="<your-api-token>"
DEVICE_ID=<your_device_id>
```

### 2. Start the endurance test

To start the test in the background, run the following command:

```bash
task start-endr
```

This will:

- Create a `results` directory if it doesn't exist.
- Start the `run_endurance_test.sh` script in the background using `nohup`.
- Redirect all output to a log file in the `results` directory (e.g., `results/endurance_test_2023_1027_1530_00.log`).
- Create an `endurance_test.pid` file containing the process ID of the test script.

If a PID file already exists and the process is still alive, the task exits with an error to prevent a duplicate launch. If the PID file is stale (the process no longer exists), it is cleaned up automatically before starting.

### 3. Check the status

To check if the test is running, use:

```bash
task status-endr
```

This prints the process information if the test is active, and cleans up the PID file automatically if the process has exited.

### 4. Stop the endurance test

To stop the test, run:

```bash
task stop-endr
```

This will kill the process and remove the `endurance_test.pid` file.

### 5. Run a single iteration (for debugging)

To run the scenario exactly once without looping:

```bash
task runn-endr
```

### 6. View logs

The logs are stored in the `results` directory. You can view the latest log file to see the test progress.

```bash
tail -f results/$(ls -t results | head -n1)
```

Each log contains per-iteration separators (`========================================`), start/end timestamps, elapsed time, the wait time until the next iteration, and the verbose `runn` output (per-step requests/responses and assertion results).

## Pass / Fail Criteria

- **Pass**: All assertions above succeed and `runn` exits with code `0`.
- **Fail**: Any assertion fails, an HTTP error occurs, or the job does not reach a terminal state within 20 polls. A failure does **not** stop the loop; it is recorded in the log and the next iteration starts on schedule. Aggregating the failure rate and failure modes from the logs is how long-running stability is evaluated.
